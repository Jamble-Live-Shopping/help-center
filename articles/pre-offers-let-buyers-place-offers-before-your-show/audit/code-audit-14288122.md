## Notes for v2 revamp

- The iOS source code uses `String(localized: "Pre-Bid")` and `"Bid"` heavily; ALL of these resolve to user-facing `Pre-Offer` / `Pré-oferta` and `Offer` / `Oferta` via xcstrings. The source key `Pre-Bid` MUST NOT leak into user-facing body or mockups. Article body uses **Pré-oferta** (pt-BR) and **Pre-Offer** (EN) exclusively. Mockups follow the same rule.
- Internal Swift error string `"You can not use prebid if you have more than one quantity"` includes lowercase `prebid`; this is shown verbatim only in a debug-style toast and is not the article's referenced surface. Article does not reproduce that exact string; the `regex:\bPre-Bid\b` ban targets the hyphenated source key only.
- `Auction` is the Swift enum case `.AUCTION` AND the xcstrings source key, but the user-facing translation is `Real Time Offer` / `Oferta em tempo real`. Rule 2c bans `auction` in article body. The article uses **Real Time Offer** (EN) / **Oferta em tempo real** (pt-BR) only.
- Screen 1 (upcoming-show product card) uses the real iOS `pre_bid_icon.imageset/pre_bid_icon.svg` SVG inlined verbatim with `fill="#FFFFFF"` adjustment (asset uses `fill="#7E53F8"` but the iOS render tints with `.alwaysTemplate` to `tintColor = .white` against the dark button background, per CalendarProductCell.swift:209-218).
- The Pre-Offer modal (PreBidViewController) renders a `ShowSlideControl(type: .preBid)` with a "Slide to Bid" gesture. The mockup uses a static rendering of the slide control with the chevron handle in start position and the button label in center, faithful to the iOS UI but not animated.
